   ```lua
       { addr = 0x030027BE, name = "STRUCT_80",   note = "0x80 struct offset" },
       { addr = 0x030027DE, name = "STRUCT_A0",   note = "0xA0 struct offset" },

       { addr = 0x0300283E, name = "STRUCT_100",  note = "0x100 struct offset" },
   }
   ```

2. Also scan a wider range around the known P1 HP area. Add a broad scan mode that checks every byte from 0x03002700 to 0x03002900 and from 0x03004C00 to 0x03004D00, tracking which addresses have u8 values in range [0, 255] that change during gameplay.

3. On each frame callback, read all candidate addresses (as u8, matching P1 HP format). Compare to previous frame values. When a value CHANGES, log to console:


4. Every 60 frames, print a summary table showing current values of all candidates.

5. Also read and display the KNOWN P1 HP (0x0300273E) as a reference -- when P1 takes damage, this should change, confirming the script is working.

6. Include clear usage instructions in the file header:

   ```
   -- Usage: mgba-qt --script lua/tools/p2_hp_scanner.lua rom.gba
   -- Then play the game via noVNC and attack P2. Watch console output
   -- for which candidate address decreases when P2 takes damage.
   ```

7. Track change counts for each address. After 300 frames, print a "likely P2 HP" recommendation based on which address changed in a pattern consistent with taking damage (decreasing, staying in 0-255 range).

**Part B: Save State Creation Guide**

Create `docs/SAVE_STATE_GUIDE.md` documenting how to create a proper fight-start save state:

1. **Why it matters**: Training needs a deterministic starting point with both players at full HP in an active fight. The current save state has P1 HP=44, P2 HP=72, round_state=71 which is likely not an active fight.

2. **Prerequisites**: Docker container running with noVNC enabled, ROM file present.

3. **Step-by-step procedure**:
   - Start the Docker container with noVNC
   - Open browser to http://localhost:6080/vnc.html
   - In the mGBA window, load the ROM if not auto-loaded
   - Navigate: Title Screen -> VS Mode (or Story Mode) -> Select character -> Start fight
   - Wait until the fight is actively in progress (both HP bars full, characters on screen)
   - In the mGBA scripting console (or via a helper script), run: `emu:saveStateFile("/data/savestates/fight_start.ss0")`
   - Alternatively, use mGBA's built-in save state feature (Shift+F1 to save to slot 1)

4. **Verification**: After saving, load the state and check memory values:
   - P1 HP (0x0300273E) should be 176 (full health) or near it
   - round_state (0x03002826) should indicate active gameplay
   - Both characters should be visible and controllable

5. **Troubleshooting**: What to do if HP values are 0, if game is in a menu, etc.

**Part C: Known Issues Document**

Create `docs/KNOWN_ISSUES.md` documenting:

1. **FPS Uncapping (BLOCKED)**: mGBA Qt runs at 60fps. Attempts to uncap via `-C fpsTarget=9999` and qt.ini both broke script loading. The training loop works at 60fps but is slower. 1800 frames = 30s real time per evaluation. With 30 genomes, ~15 min per generation. Acceptable for development, may need headless-mode solution for production training.

2. **P2 HP Address Unverified**: Address 0x03004C30 is from old VBA code and not verified. Use `lua/tools/p2_hp_scanner.lua` to identify the correct address. Until verified, fitness scores involving P2 damage may be inaccurate.

3. **Save State Validity**: Current save state may not be in an active fight. See `docs/SAVE_STATE_GUIDE.md` for recreation procedure.

4. **mGBA console:log() doesn't reach Docker stdout**: All logging goes to file at `output/training.log`. This is by design (documented in project memory).

5. **round_state semantics inverted**: 0=instant win trigger, non-zero=normal gameplay. HP-based round detection is used instead. This is a known architectural decision, not a bug.
