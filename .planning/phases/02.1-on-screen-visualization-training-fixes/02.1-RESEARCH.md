**Researched:** 2026-03-07
**Domain:** mGBA Lua Scripting (Canvas/Painter API), NEAT visualization, GBA memory forensics, Docker build configuration
**Confidence:** HIGH

## Summary

This phase addresses urgent work discovered during Phase 02 execution: implementing a MarI/O-style neural network overlay using mGBA's Painter API, fixing training pipeline bugs, and resolving unverified memory addresses that affect fitness calculation accuracy.

The mGBA master branch (which the project already builds from) provides a full Painter API with `drawRectangle`, `drawLine`, `drawCircle`, `drawText`, color control, and alpha blending -- sufficient to replicate MarI/O's neural network visualization. The API chain is `canvas:newLayer(w,h)` to get a layer, `image.newPainter(layer.image)` to get a painter, then draw calls followed by `layer:update()`. Text rendering requires FreeType (`libfreetype-dev` in builder, `libfreetype6` in runtime), which is currently missing from the Dockerfile but is ON by default in mGBA's cmake. The project already has a complete bitmap font library (`lua/vis/pixel_draw.lua`) as a fallback for environments without FreeType.

Training pipeline fixes are well-scoped: the bestFitness initialization bug is already fixed (`-math.huge`), the save state needs manual recreation via noVNC, and P2 HP address verification requires a RAM scanner script. FPS uncapping is BLOCKED due to Qt frontend limitations.

**Primary recommendation:** Implement neural network overlay using mGBA Painter API with FreeType text, falling back to pixel_draw.lua bitmap font where FreeType is unavailable. Fix Dockerfile, verify P2 HP address, and document save state creation procedure.

### Pattern 1: Canvas Overlay Initialization

**What:** Lazy-initialize a canvas overlay layer with Painter, handling graceful fallback.
**When to use:** Every vis module that draws on screen.
**Example:**
```lua
-- Source: mGBA dev scripting docs (mgba.io/docs/dev/scripting.html)
-- Verified pattern from input-display.lua sample script
local state = nil

local function init()
    if state then return state end
    if type(canvas) ~= "userdata" and type(canvas) ~= "table" then
        state = { available = false }
        return state
    end
    local ok, err = pcall(function()
        local w, h = 132, 160  -- left panel for network display
        local layer = canvas:newLayer(w, h)
        layer:setPosition(0, 0)
        local painter = image.newPainter(layer.image)
        -- Try FreeType font
        local hasText = false
        local fontPaths = {
            "/opt/mgba/share/mgba/scripts/demos/SourceSans3-Regular.otf",
            "/usr/share/fonts/truetype/dejavu/DejaVuSansMono.ttf",
        }
        for _, fp in ipairs(fontPaths) do
            local f = io.open(fp, "r")
            if f then f:close(); painter:loadFont(fp); painter:setFontSize(8); hasText = true; break end
        end
        state = { available = true, layer = layer, painter = painter, hasText = hasText, w = w, h = h }
    end)
    if not ok then
        state = { available = false }
    end
    return state
end
```

### Pattern 2: MarI/O Neural Network Display Layout

**What:** A three-column layout showing input neurons (left), hidden neurons (center), output neurons (right) with weighted connections.
**When to use:** The primary network_display.lua visualization.

MarI/O's displayGenome function uses this layout:
- **Input column** (x~50): One node per game input, vertically spaced. For this project: 9 inputs (8 game state + bias).
- **Hidden column** (x=90..220): Spring-layout positioned nodes. 4 iterations of attraction: `c1.x = 0.75*c1.x + 0.25*c2.x`. Nodes clamped to center region.
- **Output column** (x~220): One node per button (8 outputs), vertically spaced with button labels.
- **Connections:** Lines colored by weight sign (green=positive, red=negative), opacity by activation magnitude.
- **Node rendering:** Small filled circles/boxes, grayscale based on activation value.

For GBA screen (240x160), the network panel should be positioned as an overlay OUTSIDE or BESIDE the game view. The canvas can extend beyond `screenWidth`/`screenHeight`. Alternatively, a semi-transparent overlay on the left side of the game screen works.

**Adapted layout for this project:**
```
Network panel: 132px wide x 160px tall (left side)
  Input nodes:   x=10,  y=10..150, 9 nodes spaced ~15px apart
  Hidden nodes:  x=40..90, spring-layout
  Output nodes:  x=110, y=10..130, 8 nodes spaced ~15px apart
  Labels:        Input names left-aligned, output button names right-aligned
```

### Pattern 3: Frame-Throttled Overlay Updates

**What:** Update visual overlays every N frames to reduce performance impact.
**When to use:** Always -- drawing every single frame wastes cycles.
**Example:**
```lua
local draw_counter = 0
local DRAW_INTERVAL = 5  -- update every 5 frames (~12 FPS visual update at 60 FPS game)

callbacks:add("frame", function()
    trainer:tick()
    draw_counter = draw_counter + 1
    if draw_counter % DRAW_INTERVAL == 0 then
        networkDisplay.draw(currentGenome)
    end
end)
```

### Pattern 4: Dual-Renderer Architecture

**What:** Attempt Painter API first; fall back to pixel_draw.lua bitmap rendering.
**When to use:** When the code must work both inside Docker (with FreeType) and locally (possibly without).
**Example:**
```lua
local function drawLabel(painter, pd, img, text, x, y, color, usePainter)
    if usePainter then
        painter:setFillColor(color)
        painter:drawText(text, x, y)
    else
        pd.drawText(img, x, y, text, color)
    end
end
```

### Anti-Patterns to Avoid

- **Drawing every frame:** The overlay should update every 3-5 frames, not every frame. Drawing is expensive relative to the training loop tick.
- **Re-initializing overlay each frame:** Canvas layers and painters must be created once and reused. `canvas:newLayer()` allocates resources; calling it repeatedly will leak.
- **Blocking on font load failure:** If FreeType is not available, `painter:loadFont()` will error. Always wrap in pcall and fall back gracefully.
- **Using image:setPixel for bulk drawing when Painter is available:** The Painter API draws shapes in native C code. Using pixel_draw.lua's setPixel loop is 10-100x slower for the same shapes.
- **Allocating new images per frame:** `image.new()` allocates memory. Create the image once and clear/redraw it.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Drawing rectangles/lines/circles | setPixel loops | `painter:drawRectangle/drawLine/drawCircle` | Native C implementation, order of magnitude faster |
| Font rendering | Bitmap font engine | `painter:drawText()` with FreeType | Proper kerning, anti-aliasing, scalable sizes |
| Canvas overlay management | Manual screen buffer | `canvas:newLayer()` + `layer:update()` | Handles compositing, positioning, alpha blending |
| Weight-based colors | Manual RGB math | `pixel_draw.weightColor()` | Already implemented with MarI/O sigmoid convention |
| Node activation colors | Manual grayscale calc | `pixel_draw.nodeColor()` | Already implemented with activation-to-grayscale mapping |

**Key insight:** The pixel_draw.lua library is already built and battle-tested as a fallback, but the Painter API should be the primary renderer for performance. pixel_draw.lua should only be used for bitmap text when FreeType is unavailable.

## Common Pitfalls

### Pitfall 1: FreeType Not Compiled Into mGBA

**What goes wrong:** `painter:drawText()` silently fails or throws an error if mGBA was not built with `USE_FREETYPE=ON`.
**Why it happens:** The current Dockerfile does not include `libfreetype-dev` in the builder stage.
**How to avoid:** Add `libfreetype-dev` to builder apt-get (line 19) and `libfreetype6` to runtime apt-get (line 53). Verify with `cmake` output showing `Font rendering: ON`.
**Warning signs:** `painter:loadFont()` throws an error; `painter:drawText()` renders nothing.

### Pitfall 2: Canvas Layer Limit

**What goes wrong:** Creating more than 10 overlay layers causes undefined behavior.
**Why it happens:** mGBA supports OVERLAY0 through OVERLAY9 (10 layers maximum).
**How to avoid:** Use at most 2-3 layers: one for network display, one for HUD, one for debug. Re-use layers by clearing and redrawing rather than creating new ones.
**Warning signs:** Layers stop appearing or overwrite each other.

### Pitfall 3: Color Format Confusion

**What goes wrong:** Colors appear wrong (red vs blue swapped, wrong alpha).
**Why it happens:** mGBA uses `0xAARRGGBB` format. Some documentation confusingly uses `0xAABBGGRR`.
**How to avoid:** Always use the ARGB format: `0xFF00FF00` = opaque green. The pixel_draw.lua `rgba(r, g, b, a)` helper handles this correctly.
**Warning signs:** Green connections appear red, or vice versa.

### Pitfall 4: Unverified P2 HP Address

**What goes wrong:** Fitness calculation uses wrong P2 HP values, producing meaningless fitness scores.
**Why it happens:** P2 HP at `0x03004C30` is from old VBA code, never verified against the actual game.
**How to avoid:** Write a RAM scanner that watches candidate addresses during gameplay and identifies which one decreases when P2 takes damage. The candidate addresses are documented in `docs/MEMORY_MAP.md`.
**Warning signs:** P2 HP never changes during fights; all genomes get identical fitness; fitness is always -1.

### Pitfall 5: Save State in Non-Interactive Game State

**What goes wrong:** Training runs but no learning happens because the game is not in an active fight.
**Why it happens:** The current `fight_start.ss0` save state has P1 HP=44, P2 HP=72, round_state=71 -- the game appears to be in a non-interactive state (menu, cutscene, or post-fight).
**How to avoid:** The user must manually navigate to an active VS Mode fight via noVNC and create a new save state when both players have full HP and the fight is actively in progress.
**Warning signs:** HP values don't change during training; round_state is non-zero at start.

### Pitfall 6: Overlay Performance on Emulated Frames

**What goes wrong:** Training slows significantly when overlay drawing is active.
**Why it happens:** Each overlay draw call costs time, and 60 overlay updates per second compounds.
**How to avoid:** Throttle overlay updates to every 3-5 frames. Use Painter API (native C) instead of pixel_draw.lua (Lua loops). Clear and redraw only changed regions if possible.
**Warning signs:** FPS drops below 50 when overlay is enabled; training takes 2x longer.

### Drawing Network Connections (adapted from MarI/O displayGenome)

```lua
-- Source: Adapted from SethBling MarI/O neatevolve.lua displayGenome()
-- Weight-based coloring: green=positive, red=negative, opacity by activation
local function connectionColor(weight, sourceActive)
    local mag = math.abs(weight)
    local sigmoid = 2.0 / (1.0 + math.exp(-4.9 * mag)) - 1.0
    local intensity = math.floor(sigmoid * 0x80)
    local alpha = sourceActive and 0xA0 or 0x20
    if weight > 0 then
        return alpha * 0x1000000 + (0x80 - intensity) * 0x100  -- green channel
    else
        return alpha * 0x1000000 + (0x80 - intensity) * 0x10000  -- red channel
    end
end

-- Draw connection line
painter:setStrokeColor(connectionColor(gene.weight, neuronActive))
painter:setStrokeWidth(1)
painter:drawLine(srcX, srcY, dstX, dstY)
```

### Spring Layout for Hidden Nodes (from MarI/O)

```lua
-- Source: Adapted from MarI/O displayGenome() spring layout
-- 4 iterations of force-directed positioning
for _ = 1, 4 do
    for _, gene in ipairs(genome.genes) do
        if gene.enabled then
            local c1 = cells[gene.into]
            local c2 = cells[gene.out]
            if c1 and c2 then
                -- Only move hidden nodes (not input or output)
                if gene.into > Config.Inputs and gene.into <= Config.MaxNodes then
                    c1.x = 0.75 * c1.x + 0.25 * c2.x
                    if c1.x < 40 then c1.x = 40 end
                    if c1.x > 90 then c1.x = 90 end
                end
                if gene.out > Config.Inputs and gene.out <= Config.MaxNodes then
                    c2.x = 0.75 * c2.x + 0.25 * c1.x
                    if c2.x < 40 then c2.x = 40 end
                    if c2.x > 90 then c2.x = 90 end
                end
            end
        end
    end
end
```

### RAM Scanner for P2 HP Verification

```lua
-- Script to identify P2 HP address by watching for values that decrease
-- when P2 takes damage. Run in mGBA scripting console during active fight.
local candidates = {
    0x03004C30,  -- old VBA code address
    0x030027B2,  -- struct offset hypothesis (0x74 struct size)
    0x030027BE,  -- struct offset hypothesis (0x80 struct size)
    0x030027DE,  -- struct offset hypothesis (0xA0 struct size)
    0x030027FE,  -- struct offset hypothesis (0xC0 struct size)
    0x0300283E,  -- struct offset hypothesis (0x100 struct size)
}
local prev = {}
for _, addr in ipairs(candidates) do prev[addr] = emu:read8(addr) end

callbacks:add("frame", function()
    for _, addr in ipairs(candidates) do
        local val = emu:read8(addr)
        if val ~= prev[addr] then
            console:log(string.format("CHANGED: 0x%08X  %d -> %d", addr, prev[addr], val))
            prev[addr] = val
        end
    end
end)
```

## Open Questions

1. **P2 HP address verification**
   - What we know: P1 HP at `0x0300273E` is verified. P2 HP at `0x03004C30` is from old VBA code, 9,458 bytes from P1.
   - What's unclear: Whether `0x03004C30` is correct, or whether P2 struct follows a predictable offset from P1.
   - Recommendation: Write a RAM scanner script (example in Code Examples), run during active fight via noVNC, and identify the correct address by observing value changes when P2 takes damage. This requires user interaction and cannot be fully automated.

2. **Save state validity**
   - What we know: Current `fight_start.ss0` has P1 HP=44, P2 HP=72, round_state=71 -- likely not an active fight.
   - What's unclear: Whether the current state allows any meaningful training at all.
   - Recommendation: Document a clear procedure for the user to create a proper save state via noVNC. The save state must be captured during an active VS Mode fight with both players at full HP.

3. **FPS uncapping**
   - What we know: mGBA Qt runs at 60fps. Attempts to uncap via `-C fpsTarget=9999` and qt.ini both broke script loading.
   - What's unclear: Whether the `emu` API has a fast-forward mode, or whether a future mGBA release will support headless uncapping.
   - Recommendation: Accept 60fps limitation for now. Mark as BLOCKED. The training loop is functional at 60fps, just slower. 1800 frames = 30 seconds real time per evaluation. With 30 genomes, that's ~15 minutes per generation. This is acceptable for initial development.

4. **Overlay positioning strategy**
   - What we know: GBA screen is 240x160. The canvas can create layers of any size positioned anywhere.
   - What's unclear: Whether the overlay should be OVER the game (semi-transparent) or BESIDE it (extending the canvas area).
   - Recommendation: Use semi-transparent overlay ON the left side of the game screen (matches MarI/O style). This keeps the noVNC viewport compact and doesn't require resizing the mGBA window.
