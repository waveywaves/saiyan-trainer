Create `lua/utils.lua` with shared helper functions:
- `utils.formatHex(value)` -- formats a number as "0xNNNN" hex string

Create `lua/memory_map.lua` as the centralized memory address configuration file. This is the ONLY file where memory addresses live -- no other Lua file should contain hardcoded addresses.

Structure:
1. Define `DOMAIN = "System Bus"` at the top (per research: always use System Bus to match cheat database conventions, avoiding Pitfall 4 IWRAM/System Bus confusion)
2. Define the `MemoryMap` table with entries for every game state variable. Each entry is a table with fields: `addr` (hex address), `size` (bytes: 1, 2, or 4), `type` (string: "u8", "u16_le", "s16_le", "u32_le"), `desc` (human-readable description), `verified` (boolean, false for all placeholders)

Required entries (placeholder addresses using 0x03000000 range -- user fills in real addresses later via BizHawk RAM Search):
- `p1_health`: addr=0x03002700, size=2, type="u16_le", desc="P1 Health (expected range 0-100 or 0-1000)"
- `p1_ki`: addr=0x0300274A, size=2, type="u16_le", desc="P1 Ki Energy (known seed from CodeBreaker 8300274A)", verified=true (this one has a cheat code seed)
- `p1_x`: addr=0x03002710, size=2, type="s16_le", desc="P1 X Position (signed, increases moving right)"
- `p1_y`: addr=0x03002712, size=2, type="s16_le", desc="P1 Y Position (signed, changes during jumps/flight)"
- `p1_state`: addr=0x03002720, size=1, type="u8", desc="P1 Attack/Animation State (state machine byte)"
- `p2_health`: addr=0x03002800, size=2, type="u16_le", desc="P2 Health"
- `p2_ki`: addr=0x03002802, size=2, type="u16_le", desc="P2 Ki Energy"
- `p2_x`: addr=0x03002810, size=2, type="s16_le", desc="P2 X Position"
- `p2_y`: addr=0x03002812, size=2, type="s16_le", desc="P2 Y Position"
- `p2_state`: addr=0x03002820, size=1, type="u8", desc="P2 Attack/Animation State"
- `round_state`: addr=0x03002826, size=1, type="u8", desc="Round/match outcome state (known seed from CodeBreaker 33002826)", verified=true
- `timer`: addr=0x03002830, size=2, type="u16_le", desc="Match timer countdown value"

3. Add `MemoryMap.read(entry)` function that dispatches to the correct `memory.read_*` call based on `entry.type`, always passing `DOMAIN`. Supported types: u8, u16_le, s16_le, u32_le.

4. Add `MemoryMap.readAll()` function that returns a table of {name = value} for every entry in MemoryMap (skipping non-table entries like functions).

5. Add `MemoryMap.getUnverified()` function that returns a list of entry names where `verified == false` -- helps the user track which addresses still need real values.

6. Add a comment block at the top of `memory_map.lua` explaining:
   - This file is the ONLY place where memory addresses are defined
   - Placeholder addresses must be replaced with real addresses found via BizHawk RAM Search
   - All addresses use System Bus domain (matching cheat code databases)
   - Refer to docs/MEMORY_MAP.md for discovery instructions
   - The `verified` field tracks which addresses have been confirmed

Use Lua module pattern: `local MemoryMap = {}` at top, `return MemoryMap` at bottom.

Create `lua/memory_reader.lua` -- a diagnostic BizHawk Lua script that reads ALL game state from memory and displays it as an on-screen overlay. This script is meant to be loaded in BizHawk during a fight to verify that memory addresses are correct.

Structure:
1. Require `memory_map` module (use BizHawk-compatible require: `local mm = dofile("lua/memory_map.lua")`)
   NOTE: BizHawk's Lua does not use standard `require` paths. Use `dofile` with a path relative to BizHawk's working directory, OR use a path relative to the script's own location.
2. Require `utils` module similarly.
3. Define `displayOverlay(state)` function that uses `gui.text(x, y, text)` to render each state value on the BizHawk screen. Layout:
   - Left column (x=10): P1 values (health, ki, x, y, state)
   - Right column (x=150): P2 values
   - Bottom row (y=140): Round state, timer
   - Color code: verified addresses in green, unverified in yellow (use `gui.text(x, y, text, color)`)
4. Define `logToConsole(state)` function that prints all values to BizHawk's Lua console using `console.log()`, formatted as "P1 Health: 100 | P2 Health: 95 | Ki: 50 | Timer: 99"
5. Main loop: `while true do` -> read all state via `mm.readAll()` -> `displayOverlay(state)` -> `logToConsole(state)` (throttled to every 60 frames to avoid console spam) -> `emu.frameadvance()` -> `end`
6. Add a startup banner: `console.log("=== Saiyan Trainer Memory Reader ===")` and list unverified addresses using `mm.getUnverified()`.

Create `docs/MEMORY_MAP.md` -- human-readable reference documenting every memory address:

Structure:
1. Header explaining this is the memory map for DBZ: Supersonic Warriors (USA) on GBA
2. Table with columns: Name, Address (System Bus), Size, Type, Description, Verified, Notes
3. All 12 entries from memory_map.lua
4. Section "How to Discover Addresses" with step-by-step RAM Search instructions (from research):
   - Finding health: search for decreasing 16-bit value when hit
   - Finding ki: verify known 0x0300274A seed
   - Finding position: search for values that change with movement direction
   - Finding timer: search for steadily decreasing value during fight
   - Finding attack state: search for values that change during attacks (hardest)
   - Finding round state: observe 0x03002826 during round transitions
5. Section "Known Address Seeds from Cheat Databases":
   - 0x0300274A: Ki energy (CodeBreaker 8300274A 6400)
   - 0x03002826: Round outcome (CodeBreaker 33002826 0000)
6. Section "Address Conventions":
   - All addresses use System Bus domain
   - IWRAM range: 0x03000000-0x03007FFF (32KB)
   - P1/P2 likely at fixed offset (discover both independently, then verify offset)
7. Section "After Discovery" explaining how to update memory_map.lua with real addresses and set verified=true
