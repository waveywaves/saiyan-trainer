    Create 4 Tekton Task definitions and the Pipeline that wires them together, plus a PipelineRun template for manual triggering. All resources use `apiVersion: tekton.dev/v1`.

    **setup-generation.yaml** (`k8s/tekton/tasks/setup-generation.yaml`):
    Task that prepares the workspace for training. If `resume-from` param is non-empty, pull checkpoint from SeaweedFS.
    - params: `run-id` (string), `resume-from` (string, default "")
    - results: `population-initialized` (string, "true" or "false")
    - workspace: `workspace`
    - Step `fetch-checkpoint` (image: `minio/mc:latest`):
      - Set mc alias to SeaweedFS: `mc alias set swfs http://seaweedfs-s3:8333 "$ACCESS_KEY" "$SECRET_KEY" --api S3v4`
      - If resume-from is non-empty: `mc cp "swfs/$(params.resume-from)" $(workspaces.workspace.path)/checkpoint/` and write "true" to results
      - Else: write "false" to results and create empty checkpoint dir
      - env vars ACCESS_KEY and SECRET_KEY from secretKeyRef `seaweedfs-creds`

    **run-training.yaml** (`k8s/tekton/tasks/run-training.yaml`):
    Task that runs BizHawk NEAT training inside the container.
    - params: `generations` (string, default "50")
    - results: `final-generation` (string), `best-fitness` (string)
    - workspace: `workspace`
    - Step `train` (image: `saiyan-trainer/bizhawk:latest`):
      - Script that starts BizHawk with Xvfb for training:
        ```bash
        #!/bin/bash
        set -euo pipefail
        export DISPLAY=:99
        Xvfb :99 -screen 0 1024x768x24 &
        sleep 2
        # Run training with the Lua script
        /opt/bizhawk/EmuHawkMono.sh --chromeless \
          --lua="$(workspaces.workspace.path)/lua/train.lua" \
          --load-state="$(workspaces.workspace.path)/savestates/fight-start.state" \
          "$(workspaces.workspace.path)/roms/rom.gba"
        # Extract results from training output
        cat $(workspaces.workspace.path)/output/final-generation.txt \
          | tee $(results.final-generation.path)
        cat $(workspaces.workspace.path)/output/best-fitness.txt \
          | tee $(results.best-fitness.path)
        ```
      - Note: The BizHawk container image (from Phase 3) already has Xvfb, mono, etc. But since this runs as a Tekton Task step (not using s6-overlay entrypoint), we start Xvfb manually in the script. Alternatively, if the Phase 3 container uses s6-overlay as ENTRYPOINT, the task may need to override the entrypoint. Add a comment noting this may need adjustment based on Phase 3 container design.
    - timeout on this task in pipeline.yaml MUST be "0" (no timeout for training).

    **evaluate-champion.yaml** (`k8s/tekton/tasks/evaluate-champion.yaml`):
    Task that loads the best genome and runs automated fights against CPU.
    - params: none (reads champion from workspace)
    - results: `best-fitness` (string), `win-rate` (string)
    - workspace: `workspace`
    - Step `evaluate` (image: `saiyan-trainer/bizhawk:latest`):
      - Script that starts Xvfb, runs BizHawk with an evaluation Lua script that plays N fights and records wins:
        ```bash
        #!/bin/bash
        set -euo pipefail
        export DISPLAY=:99
        Xvfb :99 -screen 0 1024x768x24 &
        sleep 2
        /opt/bizhawk/EmuHawkMono.sh --chromeless \
          --lua="$(workspaces.workspace.path)/lua/evaluate.lua" \
          --load-state="$(workspaces.workspace.path)/savestates/fight-start.state" \
          "$(workspaces.workspace.path)/roms/rom.gba"
        cat $(workspaces.workspace.path)/output/best-fitness.txt \
          | tee $(results.best-fitness.path)
        cat $(workspaces.workspace.path)/output/win-rate.txt \
          | tee $(results.win-rate.path)
        ```

    **store-results.yaml** (`k8s/tekton/tasks/store-results.yaml`):
    Task that uploads genome artifacts to SeaweedFS with metadata tagging.
    - params: `run-id` (string), `generation` (string), `fitness` (string), `win-rate` (string)
    - results: `genome-s3-path` (string)
    - workspace: `workspace`
    - Step `tag-and-upload` (image: `minio/mc:latest`):
      - Set mc alias to SeaweedFS
      - Create metadata.json sidecar with run_id, generation, fitness, win_rate, date (ISO 8601), opponent ("cpu-normal")
      - S3 path convention: `genomes/{run-id}/gen-{generation:04d}/`
      - Upload: champion.json, metadata.json, population.json to that path
      - Write the champion S3 path to results
      - env vars from seaweedfs-creds secret
    - Follow the exact pattern from RESEARCH.md Code Examples section.

    **pipeline.yaml** (`k8s/tekton/pipeline.yaml`):
    Pipeline definition wiring the 4 tasks sequentially.
    - name: `saiyan-training-pipeline`
    - params: `run-id` (string), `generations-per-batch` (string, default "50"), `resume-from` (string, default ""), `fitness-threshold` (string, default "5000")
    - workspace: `shared-workspace`
    - tasks in order:
      1. `setup-generation` -- passes run-id and resume-from
      2. `run-training` -- runAfter setup-generation, timeout "0", passes generations
      3. `evaluate-champion` -- runAfter run-training
      4. `store-results` -- runAfter evaluate-champion, passes run-id, generation from run-training results, fitness and win-rate from evaluate-champion results
    - The `finally` block will be added in Plan 02 (decide-continue task for PipelineRun chaining). Add a YAML comment: `# finally: added by 04-02 (PipelineRun chaining)`

    **pipelinerun.yaml** (`k8s/tekton/pipelinerun.yaml`):
    Manual trigger template PipelineRun:
    - name: `saiyan-training-run-001`
    - pipelineRef: `saiyan-training-pipeline`
    - params: run-id="run-001", generations-per-batch="50", resume-from="", fitness-threshold="5000"
    - workspace: `shared-workspace` bound to volumeClaimTemplate (5Gi, ReadWriteOnce)
    - timeouts.pipeline: "0" (CRITICAL -- prevents silent kills)
    - Add comment explaining that users should change the name and run-id for each new run.

    **IMPORTANT anti-patterns to avoid (from RESEARCH.md):**
    - Do NOT pass genome JSON through Tekton results (4096 byte limit). Pass file paths and scalar values only.
    - Do NOT create one Task per generation. Batch 50+ generations per Task.
    - Do NOT rely on volumeClaimTemplate PVCs for persistence -- they are deleted with the PipelineRun. That is why store-results copies to SeaweedFS.
